## Required sections

These must be present. Only the collections explicitly described as non-empty have that requirement:

- `config.rooms` (non-empty), `config.labs`, `config.courses`, `config.faculty`
- `time_slot_config.times` (every weekday has a non-empty block list), `time_slot_config.classes` (at least one
  enabled pattern)

## Common validation errors

**Missing field**

```
ValidationError: 1 validation error for SchedulerConfig
rooms: field required
```

**Bad time range string**

```
ValidationError: ... FacultyConfig
times -> MON -> 0: invalid time format; expected HH:MM-HH:MM
```

**Broken cross-reference**

```
ValidationError: ... CourseConfig
faculty -> 0: "Dr. Unknown" is not in faculty list
```

**Legacy or invalid capacity shape**

Global rooms and labs must be objects containing `name` and a positive integer `capacity`; every course also needs
a positive `capacity`. Validation locations such as `/config/rooms/0` or `/config/courses/0/capacity` identify the
exact entry to migrate. A well-formed configuration whose resources are simply too small passes validation and is
reported by feasibility diagnosis instead.

**Business rules**

```
ValidationError: Configuration validation errors:
  - Faculty "Dr. Smith" has minimum credits (8) greater than maximum credits (6)
  - Course "CS101" cannot conflict with itself
  - Courses without faculty assignments: ["CS102"]
```

**Time block**

```
ValidationError: ... TimeBlock
end: End time must be after start time
```

## Performance

Rough expectations (hardware-dependent):

- **Small** — Few courses and faculty: often sub-second to seconds.
- **Medium** — Tens of courses: seconds to minutes.
- **Large** — Many courses and faculty: minutes possible.

**Ideas to speed up or shrink search**

- Tighten faculty `times` windows.
- Reduce optional room/lab choices if not needed.
- Fewer `optimizer_flags` while debugging.
- Lower **`limit`**.

## No schedules

- Check faculty availability vs. generated time grid.
- Check room/lab lists vs. course requirements.
- Compare each section's `capacity` with the capacities of its allowed rooms and labs.
- Check required features and room/lab `times`; omission means unrestricted, but a supplied map makes missing days unavailable.
- Confirm course `modality` matches meeting `delivery` values and lab meetings remain in person.
- If only lecture-room collisions are blocking lab times, verify whether `reserve_room_during_lab` should be `false`.
- Ensure `conflicts` do not make the instance unsatisfiable.
- Use `POST /validate` for structured schema and cross-reference errors before submitting.
- Use `GET /schedules/{id}/diagnosis` for feasible candidate domains, capacity shortfalls, unsat cores, provenance, and verified rule-level repair sets.
- After generation, use `GET /schedules/{id}/audit/{index}` to inspect workloads, resource use, and enabled-objective scores.

## Incremental authoring

1. Start from the validated
   [`tests/fixtures/minimal_config.json`](https://github.com/mucsci/scheduler/blob/main/tests/fixtures/minimal_config.json).
2. Add courses one at a time and re-validate.
3. Add faculty and time patterns before enabling optimizations.

## Best practices summary

1. Grow complexity gradually.
2. Rely on validation messages — they are meant to be actionable.
3. Document institutional rules in `conflicts`, `mandatory_days`, and availability.
4. Match **credits** on courses to **class patterns** exactly.

The **Configuration overview** links both the canonical minimal fixture and the larger executable `example.json`;
use those files instead of copying an unvalidated template.
