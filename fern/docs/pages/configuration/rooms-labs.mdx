---
title: Rooms and labs
description: The config.rooms and config.labs lists and how courses reference them.
---

Under **`config`**, **`rooms`** and **`labs`** name the physical resources the solver may assign.

## Shape

```json
{
  "rooms": ["Room A", "Room B", "Room C"],
  "labs": ["Lab 1", "Lab 2"]
}
```

- **`rooms`** — Non-empty list of unique room names (strings).
- **`labs`** — List of unique lab names (strings); may be empty if you have no labs.

Room and lab names must be unique. Every name used in **courses** or **faculty preferences** must appear in these
lists.

## Course usage

Each course lists acceptable **rooms** and **labs** (see **Courses**). The solver always picks a room. A non-empty lab list requires a class pattern with exactly one `lab: true` meeting; the solver reserves one of the listed labs for that meeting. Courses with an empty lab list must use a pattern with no lab meeting.

## Practices

- Use stable, human-readable names; they appear in output and APIs.
- If a course has no lab requirement, its `lab` list can be empty.
- A lab list is a requirement, not merely a preference: include only the labs that can host the designated lab meeting.
- Keep the global lists aligned with your registrar or facilities data to avoid validation errors on unknown names.
